# clean_google

Replaces Google search links on Wikidata items with proper identifiers. For
each item in the input list, every URL statement (P854, P856, P953, P973,
P1325, P2699, P2888, P8214) whose value is a Google search page carrying an
explicit `kgmid=` parameter is converted: the URL statement is removed and a
**P646 Freebase ID** or **P2671 Google Knowledge Graph ID** statement is added
instead. Google-search URLs inside references are likewise stripped, with the
id added as a reference value where it fits. `g.co` short links are resolved
via HTTP redirect first.

**Keyword-only search URLs (no `kgmid=`) are never touched.** Resolving the
`q=` search text via the KG Search API is a relevance-ranked guess that
regularly returns the wrong entity (famous namesakes, partial matches), so
those URLs are only logged to `output/review.txt` for manual handling. No API
key is needed.

Background: <https://www.wikidata.org/wiki/Talk:Q4937233>.

## Run

```bash
# from the repo root, with PYTHONPATH=projects (VS Code sets this via .env)
python projects/clean_google/clean_google_links.py               # dry run, whole list
python projects/clean_google/clean_google_links.py --limit 5     # dry run, 5 items
python projects/clean_google/clean_google_links.py --qid Q123    # one item, even if done
python projects/clean_google/clean_google_links.py --save        # really edit
python projects/clean_google/clean_google_links.py --fetch-items # rebuild input/items.txt (qlever)
```

Input: `input/items.txt` (one QID per line, generated by `--fetch-items`).
A dry run makes no edits and records nothing in `output/`, so it never blocks
a later `--save`; it does print `REVIEW` lines for keyword-only search URLs.

Each run prints `editgroup=<id>`; every edit summary links to
<https://editgroups.toolforge.org> so the batch can be reviewed/reverted there.

## Progress tracking

No database (unlike clean_youtube — there are no caches worth persisting
across runs). State lives in text files under `output/` (gitignored), written
only by `--save` runs:

- `output/done.txt` — `QID<TAB>changed|no-change[, review (Nx)]`; skipped on
  later runs.
- `output/failed.txt` — `QID<TAB>error message`; also skipped on later runs.
  Rerun individual items with `--qid` after fixing the cause.
- `output/review.txt` — `QID<TAB>statement|reference<TAB>property<TAB>url`;
  keyword-only search URLs left in place for manual handling. The item itself
  still counts as done.

Anything transient (e.g. a g.co resolution failure) fails the item loudly and
lands it in `failed.txt` instead of silently marking it done. `output/` holds
the only record of progress: keep it (on Toolforge it lives in the tool's
`$HOME/wikidata`, which persists across jobs).

## Toolforge deployment

Same recipe as clean_youtube minus every database step; this project needs no
`.env` at all (no API keys, no database):

```bash
# 0. SSH in and become the tool
ssh login.toolforge.org
become <toolname>

# 1. Clone the repo
git clone <repo-url> $HOME/wikidata

# 2. Build the venv INSIDE a job so it matches the runtime image
toolforge jobs run venv --image python3.11 --wait \
  --command "python3 -m venv ~/venv && ~/venv/bin/pip install -r ~/wikidata/projects/clean_google/requirements.txt"

# 3. pywikibot config in $HOME/.pywikibot/ (same OAuth setup as
#    remove_import_ref), then dry-run a few items as a job and inspect ~/google.out
toolforge jobs run google --image python3.11 --wait \
  --command "bash $HOME/wikidata/projects/clean_google/toolforge_run.sh --limit 5"

# 4. Run for real
toolforge jobs run google --image python3.11 --wait \
  --command "bash $HOME/wikidata/projects/clean_google/toolforge_run.sh --save"
```

Gotchas (learned on remove_import_ref): the venv only works inside a job
(matching image), never on the bastion; invoke the launcher via `bash`
(Windows-committed .sh lacks +x); the launcher sets `PYTHONUNBUFFERED=1` so the
job's `.out` file updates live.

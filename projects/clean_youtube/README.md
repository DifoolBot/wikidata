# clean_youtube

Adds metadata qualifiers to YouTube video URLs in Wikidata statements. For each
item in the input list, every non-deprecated URL statement (P854, P856, P953,
P973, P1325, P2699, P2888, P8214) pointing at a YouTube video gets, where
missing:

| Qualifier | Source |
|---|---|
| P1476 title | videos.list snippet.title |
| P577 publication date | snippet.publishedAt |
| P2047 duration | contentDetails.duration (seconds < 1 min, else minutes) |
| P11245 YouTube handle | channels.list customUrl (without `@`) |
| P2397 channel ID | only as fallback when the channel has no handle |
| P123 publisher | Wikidata item found by handle/channel-ID SPARQL lookup |
| P407 language | snippet.defaultAudioLanguage via a code→QID map |

If an existing handle / channel ID / publisher qualifier **conflicts** with the
fetched value, the item fails with a "manual check required" error instead of
being changed. Statements qualified with P6954 = link rot are skipped.

## Run

```bash
# from the repo root, with PYTHONPATH=projects (VS Code sets this via .env)
python projects/clean_youtube/youtube_metadata.py               # dry run, whole list
python projects/clean_youtube/youtube_metadata.py --limit 5     # dry run, 5 items
python projects/clean_youtube/youtube_metadata.py --qid Q123    # one item, even if done
python projects/clean_youtube/youtube_metadata.py --save        # really edit
python projects/clean_youtube/youtube_metadata.py --fetch-items # rebuild input/items.csv (qlever)
```

Input: `input/items.csv` (one QID per line, generated by `--fetch-items`).
Needs `YOUTUBE_API_KEY` in the repo-root `.env` (Google Cloud key, restricted to
the YouTube Data API v3). A dry run makes no edits and records nothing in the
`qids` table, so it never blocks a later `--save`; the channel/publisher caches
do fill during dry runs (that is their point).

Each run prints `editgroup=<id>`; every edit summary links to
<https://editgroups.toolforge.org> so the batch can be reviewed/reverted there.

## Tracker database

Progress and caches live in a small database (tables: `qids` = per-item status,
resumable via skip-if-processed; `channel_handles` = channelId→handle cache;
`channel_publishers` = handle→publisher-QID cache).

Two backends, selected with the `WD_DB_BACKEND` env var (same pattern as
remove_sitelinks):

- **Firebird** (default, local dev): schema `schemas/youtube.sql`.
- **MariaDB** (`WD_DB_BACKEND=mariadb`, Toolforge ToolsDB): schema
  `schemas/youtube_mariadb.sql`. The Firebird maintenance procedures
  (CORRECTIONS, GET_LIST) are documented there as plain queries.

Connection details come from `channel_handles.json` next to this file
(gitignored; per-machine content — Firebird locally, MariaDB on Toolforge).
DB_USER/DB_PASSWORD are taken from `WD_DB_USER`/`WD_DB_PASSWORD` in `.env`.

`migrate_youtube_to_mariadb.py` copies the data across in two steps:
`--export` locally writes `data/youtube_dump.json`; scp that to Toolforge and
run `--import` there (see below).

## Toolforge deployment

```bash
# 0. SSH in and become the tool
ssh login.toolforge.org
become <toolname>

# 1. Clone the repo; secrets in $HOME/wikidata/.env (chmod 600):
#    YOUTUBE_API_KEY=...
#    WD_DB_USER=<user from ~/replica.my.cnf>
#    WD_DB_PASSWORD=<password from ~/replica.my.cnf>
git clone <repo-url> $HOME/wikidata

# 2. Create the ToolsDB database and load the schema (bastion, no venv needed).
#    NB: `sql tools` treats extra arguments as a query, not a database name, so
#    load the schema with plain mysql instead.
sql tools
> CREATE DATABASE <user>__youtube;
mysql --defaults-file=$HOME/replica.my.cnf -h tools.db.svc.wikimedia.cloud \
  <user>__youtube < $HOME/wikidata/schemas/youtube_mariadb.sql

# 3. Write projects/clean_youtube/channel_handles.json by hand (the interactive
#    create_config prompt cannot run inside a job):
#    { "DB_HOST": "tools.db.svc.wikimedia.cloud", "DB_NAME": "<user>__youtube" }

# 4. Build the venv INSIDE a job so it matches the runtime image
toolforge jobs run venv --image python3.11 --wait \
  --command "python3 -m venv ~/venv && ~/venv/bin/pip install -r ~/wikidata/projects/clean_youtube/requirements.txt"

# 5. Import the local data (dump scp'd to data/ in the project dir first).
#    Quick one-off, so running it on the bastion is fine; the venv works there
#    when the bastion's Python matches the job image (both 3.11 currently):
~/venv/bin/python ~/wikidata/projects/clean_youtube/migrate_youtube_to_mariadb.py --import
#    If the venv refuses to run on the bastion, get a shell inside the runtime
#    image instead:  webservice python3.11 shell  (then the same command)

# 6. pywikibot config in $HOME/.pywikibot/ (same OAuth setup as remove_import_ref),
#    then dry-run a few items as a job and inspect ~/youtube.out
toolforge jobs run youtube --image python3.11 --wait \
  --command "bash $HOME/wikidata/projects/clean_youtube/toolforge_run.sh --limit 5"

# 7. Run for real
toolforge jobs run youtube --image python3.11 --wait \
  --command "bash $HOME/wikidata/projects/clean_youtube/toolforge_run.sh --save"
```

Gotchas (learned on remove_import_ref): the venv only works inside a job
(matching image), never on the bastion; invoke the launcher via `bash`
(Windows-committed .sh lacks +x); the launcher sets `PYTHONUNBUFFERED=1` so the
job's `.out` file updates live.

## Failure handling

A failing item is recorded in `qids` with `status='failed'` and the error
message, then skipped on subsequent runs. Retry lists per error category
(private/deleted video, missing audio language, publisher conflict, unparsable
URL kinds) are the GET_LIST queries documented in the schema files; rerun
individual items with `--qid` after fixing the cause. API/quota/key errors abort
the item loudly instead of masquerading as "video deleted" — error messages
deliberately never contain the request URL, because it embeds the API key.
